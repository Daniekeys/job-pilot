**Post-11, same session — Adzuna location search bug report → two-part fix:**
- **Part 1 (bug fix):** `lib/adzuna.ts`'s `detectCountry()` only recognized 3 of Adzuna's 12 actually-supported countries (`gb`/`ca`/`au`) — every other real location (Germany, France, India, Poland, Brazil, Austria, South Africa, New Zealand) silently searched the wrong country's Adzuna index and always returned 0 results. Expanded the keyword map to all 12 (list verified via `WebSearch`/`WebFetch` against Adzuna's actual docs, not assumed). Also made blank job title fall back to `profile.jobTitlesSeeking`/`currentTitle` (developer's explicit ask).
- **Part 2 (developer follow-up, same conversation): replaced the free-text Location input with a closed Country `<select>`.** New `ADZUNA_COUNTRIES` constant in `lib/utils.ts` (single source of truth for the dropdown and server-side validation). Confirmed via `WebFetch` against Adzuna's own docs that `where` has no "remote"/"anywhere" support — **"Remote" was deliberately NOT added** as an option per the developer's explicit instruction ("if it does not [support remote], do not add it"). `searchJobs()` dropped `where` entirely — country-only filtering now, no city granularity. `lib/adzuna.ts`/`agent/adzuna.ts`/`app/api/agent/find/route.ts` all simplified accordingly; `detectCountry()` keyword-matching is now used only for the one remaining free-text case (mapping `profile.preferredLocations` to a country when the dropdown is left on "Use my profile").
- **Part 3 (real-usage bug, screenshot provided): every job insert 500'd with Postgres `23514` — `jobs_job_type_check` violated.** `agent/adzuna.ts` was inserting Adzuna's raw `contract_type` (e.g. `"permanent"`) straight into `jobs.job_type`, whose CHECK constraint only allows `'fulltime' | 'parttime' | 'contract'`. Fixed with a new `mapJobType()`: `contract_type === "contract"` → `"contract"`, `contract_time === "part_time"` → `"parttime"`, else → `"fulltime"`. Added the missing `contract_time` field to the `AdzunaJob` type (only `contract_type` was typed before). Fixed the same bug pattern that was sitting in `library-docs.md`'s example code.
- **Part 4 (same screenshot, different issue): searching "uiux" returned an unrelated "Senior Software Engineer" job.** Root cause: `category=it-jobs` was hardcoded on every search per project-overview.md's original spec, but genuine UI/UX/design postings are filed under Adzuna's separate "Creative & Design Jobs" category — they were excluded before the keyword search ever ran, and the one IT result that surfaced only matched because Adzuna's `what` is full-text (title+description), not title-only. Asked the developer to choose between widening category, keeping IT-only with better messaging, or dropping the category filter entirely — **chose to widen.** `lib/adzuna.ts`'s `searchJobs()` now fans out to `ADZUNA_CATEGORIES` (`["it-jobs", "creative-design-jobs"]`) via `Promise.all` (Adzuna's `category` param takes only one tag per request, confirmed no multi-value syntax exists) and merges/dedupes by job `id`. `results_per_page` dropped `10` → `5` per category to keep the same total job count and AI-scoring token budget as before. `project-overview.md`'s feature list line updated to match the new scope.

**Post-autocomplete, same session — "clicking a suggestion doesn't fill the title" bug report.** Developer reported clicking "Software Engineer" left the input showing "sof" (the typed text). No browser-automation tool was available by default, so this agent: (1) installed Playwright + Chromium standalone into the scratchpad dir (not the project — no package.json/lockfile changes), (2) created a temporary unauthenticated `app/test-autocomplete/page.tsx` mounting the real `SearchControls`/`JobFilters`/`JobsTable`/`JobsPagination` in the same layout as the real `/find-jobs` page (since that route requires auth this agent has no credentials for), (3) drove it with real headless Chromium — both a clean synthetic click and a slower human-like mouse-path click, both with and without a candidate fix. **Could not reproduce the failure in either case** — the input correctly updated to "Software Engineer" every time, even on the unmodified pre-fix code. Applied the fix anyway (`onMouseDown={(e) => e.preventDefault()}` on each suggestion `<button>`, stopping any focus/blur shift from ever happening when a suggestion is clicked) since it's strictly more correct regardless of root cause and is the standard pattern for this exact bug class — but **the original report is unconfirmed/unreproduced**. Deleted the temporary test route afterward; nothing in `app/test-autocomplete` should exist going forward. If this resurfaces, suspect a stale dev bundle (Fast Refresh across many edits to this exact file this session) before re-investigating the event logic — ask the developer to hard-refresh first.

**Post-autocomplete, same session — "table doesn't update after search" bug report.** Developer confirmed they were testing through the real `/find-jobs` page (not a raw API call), so the earlier architecture explanation didn't apply. Diagnosed by querying the live DB directly (`mcp__insforge__run-raw-sql`, not guessed): the "software engineer"/`us` search from the previous turn *did* save 10 real jobs with genuine per-job AI reasoning (not a scoring-fallback artifact) — but this user already had 81 saved jobs total with only 8 ever scoring ≥70, and the new 10 scored 0-58. With `JobFilters`' default sort (Match Score, descending) and 20-per-page, the new jobs were almost certainly sorted onto page 2+ by older higher-scoring jobs, or hidden by a leftover "High Match" filter — `router.refresh()` was firing correctly the whole time, the *visible* page/filter view just didn't happen to include the new rows. **Root cause was sort/pagination/filter interaction, not a refresh/data bug** — this is the second time in this conversation a "feels broken" report turned out to be filter/sort state hiding correct data (see also Part 4's category issue, a different mechanism but the same "data is right, visibility is wrong" shape). Fixed: `SearchControls.tsx`'s success path now does `router.push(\`${pathname}?sort=newest\`)` instead of a bare `router.refresh()`, resetting `q`/`match`/`page` to defaults and forcing sort to "Newest" so a search's own results always land at the top of page 1.

## Decisions made

- **State lives in URL search params, not client state** — confirmed via `/architect` then explicitly delegated to "best world-class standard" by the developer. Keeps `app/find-jobs/page.tsx` as the only place doing data fetching (code-standards.md's Server-Component-only data-fetching rule), and makes filtered views bookmarkable/shareable/refresh-safe.
- **Filtering/sorting/pagination happen at the DB layer** — confirmed the installed `@insforge/sdk` wraps `@supabase/postgrest-js` directly (its own `.d.ts` references `PostgrestFilterBuilder`), so `.or()`/`.gte()`/`.lt()`/`.range()` are all real, available methods — not a guess.
- **Search term escaping** — `escapeIlikeTerm()` backslash-escapes `%`/`_` (ilike wildcards) and the term is wrapped in double quotes inside the `.or()` string (PostgREST's documented escape for commas/parens) so user input can't break or inject extra filter conditions.
- **Only `JobFilters` is a Client Component** — `JobsPagination` stayed a Server Component; pagination is pure navigation via `next/link`, no client JS needed.
- **State-sync without `useEffect`** — `JobFilters` syncs the debounced input back to the URL's `q` value using React's "adjust state during render" pattern (compare against a tracked `syncedQuery` state each render), not `useEffect` + `setState`, because the installed ESLint config's `react-hooks/set-state-in-effect` rule errors (not warns) on that.
- **`scroll={false}`** on every `JobsPagination` link and `JobFilters`' own `router.replace()` calls — paging or filtering should never yank the viewport back to the top.
